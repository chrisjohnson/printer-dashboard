## Design: Object Selection UI, API Shape, Skipped-Objects Tracking

### Constraint: No firmware object listing
Neither Bambu nor Snapmaker exposes a list of printable objects from the firmware. Bambu MQTT *may* support an `obj_list` field on newer firmware, but `parser.go` does not parse it, and the `skip_objects` command with `obj_list` is noted in comments but not implemented. Snapmaker has no object-level API at all. **The design is firmware-agnostic**: the firmware command remains a blind "skip current object", and all object tracking is local.

### 1. UI Pattern: CSS-Only Confirmation Modal

No existing modal infrastructure in `onboarding.go`. Build a lightweight CSS-only overlay:

**Trigger**: Click skip button → opens modal (replaces fire-and-forget `cmd(id,'skip')`)

**Modal structure** (in `onboarding.go`, inline HTML):
```
<div class="skip-modal" id="skip-modal-{id}" style="display:none">
  <div class="skip-modal-overlay"></div>
  <div class="skip-modal-content">
    <h3>Skip Object</h3>
    <p>Skip the current object? This cannot be undone for this print.</p>
    <div class="skipped-list">
      <!-- populated by JS: shows objects already skipped this session -->
      <p class="skipped-count">N object(s) skipped so far</p>
    </div>
    <div class="skip-modal-actions">
      <button class="btn-skip-confirm" onclick="confirmSkip('{id}')">Skip Object</button>
      <button class="btn-skip-cancel" onclick="closeSkipModal('{id}')">Cancel</button>
    </div>
  </div>
</div>
```

**CSS additions** (inline in `onboarding.go`):
- `.skip-modal` — fixed overlay, `z-index: 1000`, flex-centered
- `.skip-modal-overlay` — semi-transparent black backdrop
- `.skip-modal-content` — white card, max-width 400px, padding
- `.skipped-list` — scrollable list of previously skipped objects (if any)
- `.btn-skip-confirm` — green primary button (reuse existing button style)
- `.btn-skip-cancel` — gray secondary button
- `@media (max-width: 768px)` — modal goes full-screen, larger touch targets

**JS changes** (in `onboarding.go`):
- `openSkipModal(id)` — fetches `GET /api/printers/{id}/skipped`, populates list, shows modal
- `confirmSkip(id)` — POSTs to `/api/printers/{id}/skip`, closes modal, refreshes skipped count
- `closeSkipModal(id)` — hides modal
- Skip button `onclick` changes from `cmd('{id}','skip')` to `openSkipModal('{id}')`

### 2. API Shape

**Existing endpoint (modify)**:
- `POST /api/printers/{id}/skip` — keep as-is for firmware command, but handler also records in tracker
  - Optional JSON body: `{ "name": "object_name" }` (name is best-effort, for display)
  - Response: `{ "status": "ok", "skipped_count": 3 }`

**New endpoint**:
- `GET /api/printers/{id}/skipped` → returns list of skipped objects for this printer
  ```json
  {
    "objects": [
      { "id": "obj-1", "name": "Part 1", "skipped_at": "2026-07-14T06:00:00Z" },
      { "id": "obj-2", "name": "Part 2", "skipped_at": "2026-07-14T06:01:00Z" }
    ],
    "count": 2
  }
  ```

**No new `ListPrintObjects` endpoint** — firmware doesn't support it. Future card when Bambu `obj_list` parsing is added.

### 3. Skipped-Objects Tracking Strategy

**Server-side in-memory tracker** — new file `internal/server/skiptracker.go`:

```go
type SkippedObject struct {
    ID        string    `json:"id"`
    Name      string    `json:"name"`
    SkippedAt time.Time `json:"skipped_at"`
}

type SkipTracker struct {
    mu      sync.RWMutex
    objects map[string][]SkippedObject // printer ID → skipped objects
}
```

Methods:
- `RecordSkip(printerID, name string)` — generates an ID (monotonic counter or UUID), appends to list
- `GetSkipped(printerID string) []SkippedObject` — returns copy
- `Clear(printerID string)` — resets list (called when print completes/cancels)

**Lifecycle**:
- Created once in `Server`, passed to handlers
- `RecordSkip` called inside `handleSkipObject` after successful firmware command
- `Clear` called when printer state transitions away from `printing` (in `updateCard()` logic or via a state-change hook)
- Not persisted across server restarts — acceptable; a print session is ephemeral

**Display in status card**:
- Below the skip button: "3 object(s) skipped" (clickable to expand list)
- In the modal: full list with object names and timestamps
- The skipped count is fetched from `GET /api/printers/{id}/skipped` on each card render/update

### 4. Mobile Support

- Modal goes full-screen on mobile: width/height 100%, border-radius 0
- Confirm/cancel buttons get min-height 44px for touch targets
- Skipped list scrollable within modal

### 6. Edge Cases & Risks

- **Rapid skip clicks**: Modal prevents double-fire (button opens modal, confirm fires once). No debounce needed.
- **State transitions mid-modal**: If print finishes while modal is open, close modal via the existing `updateCard()` state check.
- **Multiple printers**: Tracker is keyed by printer ID, so no cross-contamination.
- **Memory**: Negligible — objects per print are typically <50, each ~100 bytes.
- **No firmware feedback on skip success**: The firmware command is fire-and-forget; we trust it succeeds. If it fails, the error propagates from the handler and the modal shows an alert (reuse existing error-banner pattern).
