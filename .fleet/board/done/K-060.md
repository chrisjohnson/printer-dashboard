## Working context
<!-- curated facts a teammate picking this up needs, ~15 lines max. Bigger context
     belongs in a linked doc, not here. -->
- K-059 (commit 95b942f, pre-fleet card) already fixed one flicker cause: the
  frontend `onerror` handler in `internal/server/onboarding.go` used to set
  `display:none` on the `<img>`, causing a black flash. That's fixed — onerror
  no longer hides the image.
- Remaining/distinct flicker cause found: `internal/camera/proxy.go`
  `FrameHandler` (func starts line 257). On ANY transient upstream failure it
  falls back to `servePlaceholder()` (line 200-205), a solid-gray
  `placeholderJPEG` (line 172), served with **200 OK**. This affects:
  - RTSPS/go2rtc streams: lines 315-360 (e.g. go2rtc not ready, dial timeout,
    read error → `servePlaceholder` at lines 317, 322, 328, 340, 345, 352)
  - generic HTTP/HTTPS streams: lines 363-407 (`servePlaceholder` at 373, 379)
  - Because it's a 200 with valid JPEG bytes, the frontend's `onload` (not
    `onerror`) fires and displays it — indistinguishable from real flicker,
    and NOT covered by the K-059 frontend fix at all.
- Bambu (`bambus://`) streams do NOT have this problem: `CameraManager`/
  `FrameBuffer.Latest()` (`internal/camera/bambu_buffer.go:39-48`) retains the
  last successfully received frame indefinitely; it only returns nil (→
  placeholder) before the very first frame ever arrives. Reconnects don't
  clear it (`connectionLoop`, same file, lines 168-216).
- Frontend refresh loops: `internal/server/onboarding.go` — single-frame
  images refresh every 2s (line 1051-1057, `.camera-slot img[data-frame-url]`,
  Bambu/RTSPS/generic-HTTP snapshot mode), touchscreen MJPEG images refresh
  every 3s (line 1481-1491, `.touchscreen-img`). Neither loop is itself buggy;
  they just make the backend placeholder-on-error problem visible often.
- Existing test coverage (`internal/camera/proxy_test.go`) explicitly asserts
  placeholder-on-error behavior as correct (lines 282-407) — any fix needs to
  preserve "no frame ever received yet" → placeholder, while changing
  "had a good frame, then a transient error" → keep serving last good frame.
- Planner found a second, distinct flicker source in the same function:
  `readFirstJPEGFromMJPEG` (proxy.go:209-249) silently returns
  `placeholderJPEG` bytes on a mid-read error, and the HTTP/HTTPS branch
  (line 389) treats that return value as a *successful* frame — no
  `servePlaceholder()` call site, so Code Explorer's line-41-49 grep list
  above didn't catch it. Must fix as part of this change (see Handoff notes)
  or the new cache will get poisoned with placeholder bytes on every MJPEG
  read hiccup.

**1. New cache type (add near `placeholderJPEG`/`servePlaceholder`, ~line 197):**
```go
// lastGoodFrameCache holds the most recent successfully-fetched frame per
// stream, keyed by RTSPStreamKey (rtsps://) or the raw camera URL (http/https).
// It lets FrameHandler serve the last good frame on a transient fetch
// failure instead of falling back to the gray placeholder, mirroring the
// behavior FrameBuffer already gives Bambu cameras.
type lastGoodFrameCache struct {
    mu      sync.Mutex
    buffers map[string]*FrameBuffer
}

func newLastGoodFrameCache() *lastGoodFrameCache {
    return &lastGoodFrameCache{buffers: make(map[string]*FrameBuffer)}
}

func (c *lastGoodFrameCache) get(key string) *FrameBuffer {
    c.mu.Lock()
    defer c.mu.Unlock()
    fb, ok := c.buffers[key]
    if !ok {
        fb = NewFrameBuffer()
        c.buffers[key] = fb
    }
    return fb
}
```
No eviction — the key set is bounded by the user's configured cameras, same
as `Go2RTCManager.instances` already is. `sync` is already imported
transitively? No — proxy.go doesn't import `sync` yet, add it.

**2. Two small helpers (dedupe the header-writing that's currently repeated
3x in FrameHandler: bambus/rtsps-success/http-success):**
```go
// serveLastGoodOrPlaceholder serves fb's most recent frame if one exists,
// falling back to the gray placeholder if no frame has ever been cached for
// this stream (first request, or camera never reachable).
func serveLastGoodOrPlaceholder(w http.ResponseWriter, fb *FrameBuffer) {
    if frame := fb.Latest(); frame != nil {
        writeFrame(w, "image/jpeg", frame)
        return
    }
    servePlaceholder(w)
}

func writeFrame(w http.ResponseWriter, contentType string, body []byte) {
    w.Header().Set("Content-Type", contentType)
    w.Header().Set("Cache-Control", "no-cache")
    w.Header().Set("Content-Length", strconv.Itoa(len(body)))
    w.Write(body)
}
```
Use `writeFrame` for the existing bambus branch's frame write too (line
308-311), not just the two new paths — same bytes, same headers, no
behavior change there.

**3. `FrameHandler` (line 257): create the cache once, alongside `client`:**
`frameCache := newLastGoodFrameCache()` before `return func(w ...)`.

**4. RTSPS branch (lines 315-361):** leave the `rtspMgr == nil` early
return (line 316-319) as a bare `servePlaceholder(w)` — unchanged, matches
`TestFrameHandler_RTSPS_NoGo2RTC` and is a permanent capability gap, not a
transient failure. For every other failure branch in this block
(`rtspMgr.Start` err, `FrameURL` !ok, request-build err, `client.Do` err,
`io.ReadAll` err), replace `servePlaceholder(w)` with
`serveLastGoodOrPlaceholder(w, fb)`, where
`fb := frameCache.get("rtsps:" + streamKey)` is computed once right after
`streamKey := RTSPStreamKey(parsedURL)`. On the success path (after
`io.ReadAll` succeeds), call `fb.Update(body)` then `writeFrame(w,
"image/jpeg", body)` instead of the current inline header-writing.

**5. Generic HTTP/HTTPS branch (lines 363-407):** unknown-scheme check
(line 363-366) stays a bare `servePlaceholder(w)` — unchanged, matches
`TestFrameHandler_UnknownScheme`, permanent not transient. Compute
`fb := frameCache.get("http:" + rawURL)` right after the scheme check.
Replace `servePlaceholder(w)` at the request-build-err and `client.Do`-err
branches with `serveLastGoodOrPlaceholder(w, fb)`. On success, after the
`contentType == ""` normalization, call `fb.Update(body)` then
`writeFrame(w, contentType, body)`.

**6. Required companion fix — `readFirstJPEGFromMJPEG` (lines 209-249):**
change its signature from `([]byte)` to `([]byte, bool)`: every `return
placeholderJPEG` becomes `return nil, false`, and the success return becomes
`return jpeg, true`. In the HTTP branch's multipart handling (line 386-390):
```go
if strings.HasPrefix(contentType, "multipart/x-mixed-replace") {
    var ok bool
    body, ok = readFirstJPEGFromMJPEG(resp.Body)
    if !ok {
        serveLastGoodOrPlaceholder(w, fb)
        return
    }
    contentType = "image/jpeg"
} else {
    ...unchanged (io.ReadAll)...
}
```
Without this, a mid-read MJPEG error would call `fb.Update(placeholderJPEG)`
and permanently poison the cache with the gray placeholder, overwriting any
real last-good frame — actively worse than today for that one case. No
existing test covers `readFirstJPEGFromMJPEG` directly or this branch's
failure path (confirmed via grep), so this is a safe signature change.
